# Документация проекта

## 1. Общее описание

Веб-приложение предназначено для демонстрации учебных и практических работ. Сайт используется как витрина проектов и заданий с возможностью структурированного просмотра и дальнейшего масштабирования под API.

Проект ориентирован на **модульную архитектуру**, поддержку **TypeScript**, а также плавный переход от локальных JSON-данных к серверному API.

---

## 2. Цели проекта

- Централизованное хранение и отображение учебных и практических работ
- Удобная навигация и фильтрация
- Масштабируемая архитектура (JSON → API)
- Чистая и поддерживаемая кодовая база

---

## 3. Типы контента

> Работы добавляются **вручную в базу данных** (через админку или напрямую), без пользовательского ввода с клиентской стороны.

### 3.1 Разделы работ

- Практические работы
- Лабораторные работы
- Практические страницы

Особенности:

- Номера работ **уникальны только в рамках одного раздела**
- Работы выполняются **индивидуально**, даже если номера совпадают у разных авторов
- Автор указывается строкой (ФИО)

Каждый раздел содержит список работ с группировкой по авторам.

### 3.2 Раздел «Сайты»

Раздел предназначен для демонстрации готовых веб‑сайтов.

Для **каждого сайта** отображаются:

- Основная страница сайта (pages[] — массив объектов `{ name, url }`)
- Ссылка на GitHub‑репозиторий
- Ссылка на макет Figma (одна)
- Референсы (изображения страниц или элементов сайтов)

---

## 4. Основной функционал

### 4.1 Группировка и фильтрация

Для практических, лабораторных и учебных практических работ:

- Группировка по ФИО автора
- Фильтр по типу работы (`practice | lab | educational`)
- Фильтр по номеру работы
- Корректное отображение самостоятельных работ с одинаковыми номерами

**Примечание:** архитектура поддерживает **неопределённое количество работ на каждого автора**, поэтому фильтры должны быть динамическими.

### 4.2 Навигация

- Клиентская маршрутизация (React Router DOM)
- Раздельные страницы для каждого типа работ:
  - `/works/practice` — Практические работы
  - `/works/lab` — Лабораторные работы
  - `/works/educational` — Практические страницы
- `/sites` — Сайты
- Ссылка «Реферат» в навигации, ведущая на скачивание Word-документа.

- Детальные страницы:
  - `/works/:id`
  - `/sites/:id`

Поиск по сайту **не предусмотрен**.

---

## 5. Источник данных

### 5.1 Текущая реализация

- Данные хранятся в JSON-файлах
- JSON используется как временный источник данных

### 5.2 Проектирование под API

Планируемый тип API: **REST**

При разработке учитывается:

- Чёткое разделение data-layer
- Абстракция источника данных (JSON / API)
- Использование единых DTO-интерфейсов
- Минимальные изменения компонентов при переходе на API

Авторизация **не требуется**.

### 7.1 Подход

Используется **модульная (feature-based) архитектура**.

Каждый функциональный блок инкапсулирует:

- UI
- Логику
- Типы
- Источники данных

## 8. Типы данных (концептуально)

### 8.1 Работа

- id
- title
- workNumber
- author (ФИО строкой)
- type (practice | lab | educational)
- description
- images[] (фотографии работы)
- codeUrl? (ссылка на репозиторий или архив)
- reportUrl? (отчёт в формате Word / PDF)

Превью **не используется**.

### 8.2 Сайт

Единая сущность сайта.

- id
- title
- pages[] (массив объектов `{ name, url }`)
- repositoryUrl
- figmaUrl
- references[] (изображения)

---

## 9. UI и стилизация

- Основная тема: **тёмная**
- Общий стиль: **современный**
- Tailwind CSS используется как основной инструмент стилизации
- Компоненты проектируются переиспользуемыми
- Вся HTML-разметка должна быть **семантически корректной**:
  - `header`, `nav`, `main`, `section`, `article`, `footer`
  - корректное использование `ul / ol`, `figure / figcaption`
  - логичная иерархия заголовков `h1–h6`

- Адаптивность закладывается на уровне базовых компонентов

---

## 10. Масштабирование и развитие

Планируемые улучшения (при необходимости):

- Подключение REST API
- Асинхронная загрузка данных
- Пагинация и сортировка
- Расширенные детальные страницы работ и сайтов

Сложные механизмы (поиск, кеширование, авторизация) **не закладываются заранее**.

---

## 11. Ограничения

- Без серверного рендеринга
- Без SEO-оптимизации на первом этапе
- Без мультиязычности
- Без авторизации
